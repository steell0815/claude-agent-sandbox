# Agent Sandbox — Design Plan

**Status:** draft for review
**Date:** 2026-05-08
**Scope:** Docker-based sandbox for AI agents (Claude Code + others) with MITM egress control, payload inspection, and bounded logs.

---

## Goal

Run AI agents inside isolated Docker containers, where:

- The agent only sees the project mounted as a volume.
- The agent's home (`~/.claude` profile: settings, skills, hooks) comes from a git repo.
- Secrets are injected via env vars.
- The agent has no direct network — egress only through a MITM proxy sidecar with full payload inspection.
- All logs are rotated and bounded.

---

## Architecture

Three services in one compose stack:

| Service      | Role                                                                  |
| ------------ | --------------------------------------------------------------------- |
| `proxy-init` | One-shot. Generates CA keypair on first run if absent.                |
| `proxy`      | mitmdump with `policy.py` addon. Attached to `internal` and `egress`. |
| `agent`      | Claude Code (or other agent). Attached only to `internal`.            |

### Networks

- `internal` — `internal: true`. Agent ↔ proxy only. No external egress.
- `egress` — normal bridge. Proxy only. Outbound to host/internet.

> Earlier sketch had the proxy on `network_mode: host`. Decision: avoid it. The bridge model preserves more isolation and works fine unless something on the host needs inbound access from the proxy.

### Volumes

| Volume           | Mount in proxy            | Mount in agent          | Purpose                                        |
| ---------------- | ------------------------- | ----------------------- | ---------------------------------------------- |
| `./project`      | —                         | `/workspace` (rw)       | Project working tree (bind).                   |
| `claude-profile` | —                         | `/home/agent/.claude`   | Profile from git (named volume, init-cloned). |
| `ca`             | `/home/mitmproxy/.mitmproxy` (ro) | —                       | CA keypair. Private key never reaches agent. |
| `logs`           | `/var/log/proxy` (rw)     | `/var/log/proxy` (ro)   | Audit + rotated app logs.                      |

## MITM proxy

**mitmproxy** picked for: scriptable policy in Python, SSE support, HTTP/2 support, ergonomic.

### Policy (`policy.py`)

Responsibilities:

1. Host allowlist by regex.
2. Optional per-host method+path rules.
3. Outbound DLP regex on request bodies (AWS keys, `-----BEGIN ... PRIVATE KEY-----`, JWT-shaped tokens, etc.).
4. JSONL audit log via `RotatingFileHandler`.

Block by setting `flow.response = http.Response.make(403, ...)` from the `request` hook.

### CA management

- Generated by `proxy-init` on first run, written to `ca` volume.
- mitmproxy mounts `ca` read-only.
- Agent image baked with **public cert only** at build time (`ARG CA_CERT_PATH`).
- Private key **never** mounted to agent.

CA rotation = regenerate keypair, rebuild agent image, redeploy. Treat the keypair as a deployment secret.

## Agent image

```dockerfile
FROM node:22-bookworm-slim
ARG CA_CERT_PATH
COPY ${CA_CERT_PATH} /usr/local/share/ca-certificates/mitm.crt
RUN apt-get update \
 && apt-get install -y ca-certificates git curl python3 jq \
 && update-ca-certificates \
 && useradd -m -u 1000 agent
USER agent
WORKDIR /workspace
ENV NODE_EXTRA_CA_CERTS=/etc/ssl/certs/ca-certificates.crt \
    REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt \
    SSL_CERT_FILE=/etc/ssl/certs/ca-certificates.crt \
    GIT_SSL_CAINFO=/etc/ssl/certs/ca-certificates.crt \
    HTTP_PROXY=http://proxy:8080 \
    HTTPS_PROXY=http://proxy:8080 \
    NO_PROXY=localhost,127.0.0.1
RUN npm i -g @anthropic-ai/claude-code
ENTRYPOINT ["claude"]
```

Compose-level hardening on the agent:

```yaml
user: "1000:1000"
cap_drop: [ALL]
security_opt: [no-new-privileges:true]
read_only: true
tmpfs:
  - /tmp:size=512m,mode=1777
```

## Permissions layering

Three rings, defense in depth:

1. **Harness** — existing `settings.json` allow/deny + `validate-bash.sh` hook (already in this repo).
2. **Container** — `cap_drop: [ALL]`, read-only rootfs, non-root user, no-new-privileges.
3. **Network** — proxy allowlist + DLP.

Each ring is bypassable alone. Layered, they're meaningful.

## Logging & rotation

Two streams, two mechanisms.

### 1. Container stdout/stderr — Docker `local` driver

```yaml
x-logging: &default-logging
  driver: local
  options:
    max-size: "10m"
    max-file: "5"
    compress: "true"

services:
  proxy:
    logging: *default-logging
  agent:
    logging: *default-logging
```

~50 MB ceiling per service. Could also live in `/etc/docker/daemon.json` to apply host-wide.

### 2. Application logs (`audit.jsonl`) — `RotatingFileHandler` inside policy.py

```python
import logging, json
from logging.handlers import RotatingFileHandler

audit = logging.getLogger("audit")
audit.setLevel(logging.INFO)
h = RotatingFileHandler(
    "/var/log/proxy/audit.jsonl",
    maxBytes=50 * 1024 * 1024,
    backupCount=10,
    encoding="utf-8",
)
h.setFormatter(logging.Formatter("%(message)s"))
audit.addHandler(h)

def log(record): audit.info(json.dumps(record))
```

50 MB × 10 = 500 MB ceiling. In-process, no sidecar.

### 3. Drop mitmproxy `save_stream_file`

Append-only binary; mitmproxy holds the FD open → external rotation needs `copytruncate`, fragile. JSONL audit covers ops needs. For forensics, run a one-shot `mitmdump -w` on demand for that session only.

### 4. Bound tmpfs

`/tmp:size=512m,mode=1777` so a runaway process can't pin host RAM.

## Profile from git

Init container clones profile repo into `claude-profile` volume. Pin to a commit SHA in `.env` for reproducibility.

**Open:** clone-on-startup (fresh, requires git auth in init container) vs bake-at-build (reproducible, can lag).
**Leaning:** clone-on-startup with a deploy key. Profile evolves often; freshness wins.

**Open:** Claude Code writes to `~/.claude/sessions/`, `todos/`, `cache/`, `file-history/`, `projects/`. Two options:

- (a) Mount each as a `tmpfs` subpath — ephemeral per run, fully sandboxed.
- (b) Persist in the named volume and add periodic cleanup.

**Leaning:** (a). Resumable work isn't a goal in a sandbox. Memory (`projects/<project>/memory/`) is the one exception worth persisting — see open questions.

## IDE integration (VS Code + IntelliJ)

Both IDEs use the [devcontainers spec](https://containers.dev/), so a single `.devcontainer/devcontainer.json` per project drives both. The IDE's remote backend (VS Code Server / JetBrains backend) installs itself inside the container automatically; both attach via `docker exec`, so no exposed port is needed and `internal: true` stays intact.

**Mode:** one devcontainer **per project**. The `.devcontainer/` lives in the project repo and references the sandbox compose files by relative path (or via a `sandbox` git submodule).

**Image variant:** `agent-ide` is built as an additional **multi-stage target** in the same Dockerfile. Keeps the lean one-shot image small; the IDE-heavy variant only carries extra cost when used.

```dockerfile
FROM node:22-bookworm-slim AS agent-base
# shared: CA, certs env, agent user, claude install, proxy env

FROM agent-base AS agent
ENTRYPOINT ["claude"]

FROM agent-base AS agent-ide
USER root
RUN apt-get update \
 && apt-get install -y libxext6 libxrender1 libxtst6 libxi6 libfreetype6 libfontconfig1 procps \
 && rm -rf /var/lib/apt/lists/*
USER agent
CMD ["sleep", "infinity"]
```

**Compose split:** base `docker-compose.yml` keeps the one-shot path; `docker-compose.dev.yml` is an overlay for IDE mode. Selects the `agent-ide` target, swaps `command` to `sleep infinity`, adds named volumes for IDE state (named, **not** tmpfs — backends are hundreds of MB and re-downloading every session is wasteful).

```yaml
# docker-compose.dev.yml
services:
  agent:
    build:
      target: agent-ide
    command: sleep infinity
    volumes:
      - vscode-server:/home/agent/.vscode-server
      - jetbrains-cache:/home/agent/.cache/JetBrains
      - jetbrains-share:/home/agent/.local/share/JetBrains
volumes:
  vscode-server:
  jetbrains-cache:
  jetbrains-share:
```

**devcontainer.json (per-project, in the project repo):**

```json
{
  "name": "agent-sandbox",
  "dockerComposeFile": [
    "../sandbox/docker-compose.yml",
    "../sandbox/docker-compose.dev.yml"
  ],
  "service": "agent",
  "workspaceFolder": "/workspace",
  "remoteUser": "agent",
  "customizations": {
    "vscode": {
      "extensions": ["anthropic.claude-code"]
    }
  }
}
```

**Extensions:** allowlist-based, installed at runtime via the marketplace; pinned in `devcontainer.json` for reproducibility. Extension host runs **inside the container** so the agent's view and the IDE's view are consistent.

Proxy allowlist additions for IDE mode (gated behind a dev-mode flag in `policy.py` so the one-shot agent doesn't carry them):

- VS Code: `marketplace.visualstudio.com`, `update.code.visualstudio.com`, `*.vscode-cdn.net`, `*.gallerycdn.vsassets.io`
- JetBrains: `plugins.jetbrains.com`, `download.jetbrains.com`, `*.jetbrains.com`

**Permissions caveat (IDE mode):** integrated-terminal commands run as the `agent` user but are **not** subject to `validate-bash.sh` — that hook only fires on Claude Code's `Bash` tool calls. Container hardening + proxy still apply, so the rings shift but defense in depth holds. Worth surfacing in user-facing docs.

## Authentication

**Primary path: pre-auth on host, mount credential read-only.** Run `claude login` once on the host (browser handoff just works there) and the agent inherits the credential through a read-only bind mount. No OAuth ceremony in the container, no port forwarding, no in-container browser handoff to engineer.

```yaml
agent:
  volumes:
    - ${HOME}/.claude/.credentials.json:/home/agent/.claude/.credentials.json:ro
```

The credential is mounted **separately from the profile volume** — it must never end up in the profile git repo. Profile (settings, skills, hooks) is shared/versioned; credential is per-user and secret.

### Trade-offs to be aware of

- **Read-only blocks refresh write-back.** When Claude Code rotates its access token, it persists the new value back to the credential file — read-only makes that fail (or get silently dropped). In practice the in-memory token survives the session; long-lived containers may eventually need restart to pick up a freshly-refreshed host token. Acceptable for typical use. For multi-day sessions, consider a narrow read-write mount of *just* the credential file.
- **macOS Keychain.** Claude Code on macOS stores credentials in the Keychain by default, not as a file on disk. Two options:
  1. Force file-based storage via a config flag/env (verify exact mechanism — likely `CLAUDE_CONFIG_DIR` or a setting in `~/.claude/settings.json`) and re-login.
  2. Extract the credential at container start with a small host-side helper: `security find-generic-password -s "Claude Code" -w > ~/.claude/.credentials.json` (or whatever the keychain item is named), then mount that file. The helper runs on the host before `docker compose up`, not inside the container.
- **Trust scope.** The agent has read access to your OAuth token; treat the container with the same trust level as a process running as you on the host. Proxy + DLP still mean something against unintentional exfiltration, not against an actively malicious payload.

### Alternative: `ANTHROPIC_API_KEY` env

For accounts without OAuth (API-only billing) or CI/headless agents where there's no host-side login. Same one-line `.env` entry; no host file dependency.

### Deferred: in-container OAuth flow

Engineering the browser handoff (port-forward the localhost listener back to the host browser) is doable but adds non-trivial complexity for a corner case — user wants to OAuth from inside the container without ever logging in on the host. Skip until someone asks.

## Distribution to projects

Phased — sandbox is in flux, distribution mechanism should match the maturity stage.

### Phase 1 (now): template/scaffold script

`sandbox/scripts/init-devcontainer.sh` runs in a project root and writes `.devcontainer/devcontainer.json` from a template. The generated file references the local sandbox repo via env-var substitution so the path isn't baked in:

```json
"dockerComposeFile": [
  "${localEnv:CLAUDE_SANDBOX}/docker-compose.yml",
  "${localEnv:CLAUDE_SANDBOX}/docker-compose.dev.yml"
]
```

User exports `CLAUDE_SANDBOX=~/code/sandbox` (or wherever) once. Cheap, zero infrastructure, easy to iterate while the sandbox is changing weekly.

### Phase 2 (when stable): publish images, slim devcontainer.json

Build and push `ghcr.io/<you>/agent-sandbox-proxy:<tag>` and `ghcr.io/<you>/agent-sandbox-agent-ide:<tag>`. Project-side `devcontainer.json` references images by tag — no compose-file path resolution, no env var. Version pinning becomes "what tag did you specify."

### Avoided

- **Git submodules** — version-pin nicely but the contributor UX is bad and image tags do the same job.
- **Single global install** — every collaborator would need the same path / env var. Published images solve this without absolute-path coupling.

### Per-project policy overrides (cross-cutting)

Defaults stay tight; each project can opt into extras through a mounted file rather than rebuilding the proxy image.

`policy.py` reads two env-pointed files at startup if present:

- `ALLOWLIST_EXTRA` — additional host regexes, one per line.
- `DLP_EXTRA` — additional DLP regex patterns.

Project's `devcontainer.json` mounts them and sets the env vars:

```json
"mounts": [
  "source=${localWorkspaceFolder}/.devcontainer/allowlist.txt,target=/etc/proxy/allowlist.txt,type=bind,readonly"
],
"remoteEnv": {
  "ALLOWLIST_EXTRA": "/etc/proxy/allowlist.txt"
}
```

(The mount has to land on the proxy container, not the agent — exact wiring depends on whether devcontainers can target a non-primary service for mounts. May need a `docker-compose.override.yml` per project for the proxy mount; one of the open questions below.)

This way the published image is one-size-fits-all, and per-project relaxations are version-controlled in the project repo and reviewed like any other code change.

## Known limitations

- **Cert pinning**: Go binaries with embedded roots, `gcloud`, some mobile SDKs will fail under MITM. `npm`, `pip`, `cargo`, `git`, `curl`, `gh`, the Anthropic SDK, and Claude Code all honor the system trust store and work.
- **SSE audit granularity**: Anthropic's API uses SSE; the `response` hook fires when the stream closes. For per-chunk audit, hook `responseheaders` + `response_chunk` — doable but adds complexity.
- **CA private key**: anyone with it can forge any cert the agent trusts. Generate per-deployment, never commit, never expose to agent container.
- **DLP is best-effort**: regex catches obvious leaks (AWS keys, private keys). Not anti-exfiltration against an adversarial agent. Trust boundary, not anti-exfiltration.

## Decisions log (so far)

| Decision                                                | Why                                                                       |
| ------------------------------------------------------- | ------------------------------------------------------------------------- |
| Proxy on bridge network, not `network_mode: host`        | Preserves isolation; host network only needed if proxy must serve inbound on host iface. |
| mitmproxy over Squid                                    | Policy as Python code is reviewable, testable, versionable.                |
| `local` log driver over `json-file`                      | More efficient on disk, native compression, native rotation.               |
| In-process `RotatingFileHandler` over logrotate sidecar | One fewer container; no `copytruncate` complications.                     |
| Drop `save_stream_file`                                  | Hard to rotate; audit JSONL covers ops needs; capture full flows on demand. |
| Public CA cert baked at build time                       | Simpler than runtime injection; rotation = rebuild, which is fine.        |
| `agent-ide` as multi-stage target in same Dockerfile     | Lean one-shot path stays small; IDE extras only built when needed.        |
| Compose split: base + `docker-compose.dev.yml` overlay   | Keeps one-shot CLI usage uncluttered; IDE mode opt-in via overlay.        |
| One devcontainer per project                             | Aligns with how VS Code and JetBrains Gateway both expect to attach.      |
| IDE extension host runs inside the container             | Extensions and agent share the same filesystem/network view.              |
| Allowlist + pinned `extensions` array (not pre-bundled)  | Reproducible installs without bloating image; updates pass through proxy. |
| Named volumes for IDE state (not tmpfs)                  | VS Code Server / JetBrains backend are heavy; re-download per session is wasteful. |
| Dev-mode allowlist gated in `policy.py`                  | One-shot agent doesn't need marketplace access; smaller egress surface.   |
| Default auth = `ANTHROPIC_API_KEY` env                   | Sidesteps OAuth entirely for the sandbox use case; works headless.        |
| OAuth via IDE automatic port forwarding                  | Reuses host browser without exposing inbound network from the container. |
| Pre-auth-on-host as CI fallback                          | Refresh tokens carry the workload when no human/browser is available.    |
| Phase 1 distribution = template script + `${localEnv:CLAUDE_SANDBOX}` | Cheap, iterable while sandbox is in flux; no registry needed.    |
| Phase 2 distribution = published images + slim devcontainer.json | Version pinning via image tag; portable across teammates.        |
| Avoid git submodules for distribution                    | Image tags give the same version-pin without the contributor UX friction. |
| Per-project policy overrides via env-pointed mounted files | Defaults stay tight; relaxations are version-controlled in the project. |
